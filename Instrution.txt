Desafio: Transações, Backup e Recovery (Banco: ecommerce)

Conteúdo do repositório
- `transactions.sql`  - Exemplos de transações sem procedure.
- `procedure_transaction.sql` - Procedure com transação, savepoint e tratamento de erro.
- `sample_schema.sql` - (Opcional) script para criar esquema mínimo de teste (products, customers, orders, order_items).
- `backups/` - Pasta sugerida para colocar arquivos de backup (ex.: `ecommerce_full_backup_2026-04-09.sql.gz`).
- `README.md` - Arquivo com instruções.

Como usar os scripts SQL
1. Conecte ao MySQL como usuário com privilégios adequados.
2. (Opcional) Crie um banco de teste e rode `sample_schema.sql` para popular tabelas.
3. Execute `transactions.sql` para testar transações manuais.
4. Execute `procedure_transaction.sql` para criar a procedure:
   - `CALL sp_create_order_with_items(1, 101, 2, 49.90, 102, 1, 29.90);`

Backup com mysqldump
- Backup completo (com routines, events e triggers):
  `mysqldump -u root -p --routines --events --triggers --databases ecommerce > ecommerce_full_backup.sql`
- Backup consistente (InnoDB):
  `mysqldump -u root -p --single-transaction --routines --events --triggers --databases ecommerce > ecommerce_consistent.sql`
- Compressão:
  `mysqldump ... | gzip > ecommerce_full_backup.sql.gz`

Restore
- Restore simples:
  `mysql -u root -p < ecommerce_full_backup.sql`
- Restore de arquivo compactado:
  `gunzip < ecommerce_full_backup.sql.gz | mysql -u root -p`

Observações importantes
- Teste sempre o restore em ambiente de homologação.
- Para grandes backups, prefira armazenar em storage (S3, Azure Blob) e não no Git.
- Se incluir backups no repositório, documente o tamanho e a data no README.
